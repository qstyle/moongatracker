# Богатые этапы роадмапа (Rich Stages)

## Context

Moonga Studio ведёт продукт по роадмапу **этапов** «идея → прод». Сейчас этапы — пустые вехи: доска под этап создаётся без карточек, вики проектная (не привязана к этапу). Цель фичи — сделать этапы **каркасом-помощником**: при разворачивании этапа фаундер сразу получает доску с шаблонным бэклогом, вики-раздел этапа и подборку инструментов/ссылок под задачи этого этапа. Пройденные этапы уезжают в «Историю», не засоряя активный путь; после первичного пути фаундер добавляет свои этапы под допиливание фич.

Набор этапов определён отдельным deep-research (продукт/билд ракурс, синтез Cagan/dual-track, Double Diamond, Shape Up, Software Release Life Cycle, AWS AI-DLC): отчёт в этой сессии. Ключевой вывод: для соло это не водопад, а разделение активностей; «петля» не моделируется механикой — вместо неё пройденные этапы → История, новые этапы добавляются вручную.

## Новый набор этапов (замена дефолтов)

`buildDefaultStages` (apps/api/src/stages/default-stages.ts) переопределяется на 5 этапов (Shape влит в «Дизайн и план»):

| key | Заголовок | Цель | Гейт выхода (подсказка агента) |
|---|---|---|---|
| `discovery` | Открытие | Понять реальную проблему и пользователя | Боли повторяются в 3+ интервью, решение GO/PIVOT/KILL |
| `design` | Дизайн и план | Выбрать вариант решения + зафиксировать scope и «аппетит» | Выбран один вариант, задача «concrete enough», задан таймбокс |
| `build` | Сборка MVP | Собрать релизопригодный MVP | Feature-freeze (feature-complete) |
| `release` | Релиз | Довести до стабильного публичного релиза | alpha→beta→GA: нет showstopper-багов |
| `operate` | Эксплуатация | Мониторинг, поддержка, мелкие итерации | Финального гейта нет (конечное состояние) |

Первый этап сидируется `active`, остальные `not_started` (как сейчас). Иконки на роадмапе (roadmap.tsx STAGE_ICONS) обновить под новые ключи.

## Модель данных

- `WikiSection` — добавить `stageId String?` (nullable) + relation к Stage; `@@index([stageId])`. Разделы без stageId — обычная проектная вика (как сейчас). Миграция аддитивная.
- Никаких других изменений схемы: доски уже имеют `stageId`; «пройденность» — существующий `Stage.status='done'`; шаблоны и инструменты/ссылки — данные (seed), не таблицы.

## Разворачивание этапа

Кнопка **«Развернуть этап»** у этапа на роадмапе (для не развёрнутых). По нажатию (один транзакционный вызов API `POST /api/projects/:pid/stages/:stageId/scaffold`):
1. Создаёт **доску** под этап (Board со stageId) с 4 дефолтными колонками (как сейчас).
2. Сеет **шаблонный бэклог** — карточки из шаблона этапа в колонку «Открыто».
3. Создаёт **вики-раздел этапа** (WikiSection со stageId, заголовок = название этапа) со страницами из шаблона + страницей «Инструменты и ссылки».
Идемпотентность: если у этапа уже есть развёрнутая доска (или scaffold уже делали) — не дублировать; кнопка меняется на «Открыть доску». Разворачивать можно только дефолтные этапы (у кастомных шаблона нет — им обычное «Создать доску»).

## Шаблоны этапов (данные)

Модуль `apps/api/src/stages/stage-templates.ts`: `STAGE_TEMPLATES: Record<StageKey, StageTemplate>`, где
```ts
interface StageTemplate {
  backlogCards: { title: string; body?: string; priority?: 'urgent'|'normal'|'low' }[];
  wikiPages: { title: string; body: string }[]; // markdown
  tools: { name: string; url: string; note?: string }[];
  links: { title: string; url: string }[];
}
```
Контент (по 5 этапам): бэклог-карточки и вики-страницы берутся из deep-research (в отчёте есть «задачи доски» и «вики» по каждому этапу). `tools`+`links` — из фокусной курации/второго ресёрч-прохода под финальные этапы. «Инструменты и ссылки» рендерятся в отдельную вики-страницу этапа (markdown-список).

## Гейты — подсказки агента

Переход между этапами (кнопка «Продвинуть») двигает этап мгновенно (текущий→done, следующий→active), как сейчас в roadmap.tsx, но рядом показывается **подсказка**: критерии выхода текущего этапа (из шаблона/таблицы выше) — чтобы человек свериться перед переходом. Агент (в чате/через MCP) может напомнить критерии, но НЕ блокирует и НЕ требует апрува (proposal-гейт не используем).

## История

Когда этап переходит в `status='done'`, он вместе с досками **автоматически** уходит в свёрнутый блок **«История»** внизу роадмапа:
- Активная часть роадмапа показывает этапы со статусом `active`/`not_started`.
- Свёрнутый блок «История (N)» — done-этапы; разворот по клику показывает их с досками (доски сохраняются, кликабельны).
- Реализация — чисто клиентская группировка в roadmap.tsx по `status` (без изменения API/схемы).

## Расширение этапов

После «Эксплуатации» фаундер добавляет свои этапы кнопкой «＋ этап» (уже поддерживается — Stage CRUD). Кастомные этапы без шаблона; для них — обычное «Создать доску» (без scaffold).

## Терминология

В UI роадмапа единый термин **«этап»** (сейчас смешано «стадия»/«этап»). Проверить строки в roadmap.tsx и связанных.

## Out of scope

- Метрики/здоровье/kill-гейт (была #36, снята).
- Автоматическая проверка критериев выхода агентом (сейчас только подсказки).
- Портфель венчуров (#33).
- Циклическая механика «петля» (заменена Историей + ручным добавлением этапов).

## Verification

- `nx run-many -t typecheck` — без новых ошибок (сверх предсуществующих spec).
- `nx test api` — зелёный, включая тесты scaffold (создаёт доску+карточки+вики-раздел; идемпотентность) и WikiSection.stageId.
- `nx build api web` — зелёные.
- Ручной прогон: новый проект → роадмап показывает 5 новых этапов → «Развернуть этап» на «Открытие» создаёт доску с шаблонными карточками + вики-раздел «Открытие» с инструментами/ссылками → «Продвинуть» показывает подсказку критериев и двигает этап → пройденный этап уходит в «Историю» с доской.
