# Moongatracker — дизайн

Дата: 2026-06-29
Статус: согласован, готов к плану реализации

## 1. Цель и контекст

Self-hosted канбан, в котором **AI-агент — первоклассный участник**, а не надстройка.
Сценарий-драйвер:

> Пользователь закидывает идею → вместе с агентом (OpenClaw в мессенджере или Claude)
> разбирает её → если «выстрелила», карточка переходит в разработку → ведётся до релиза.

Доска выполняет роль **общей памяти** человека и агента: всё состояние, обсуждение и
история живут на карточке, агент читает и меняет их через API/MCP.

Окружение: тот же self-hosted сервер, где уже работают **n8n** и (планируется) **OpenClaw**.
Бюджет ресурсов жёсткий (2 vCPU / ~4 ГБ RAM на всех), поэтому стек выбран по принципу
максимальной экономии.

## 2. Принципы

- **Экономия рантайма** — один Node-процесс + встроенная SQLite; SPA без SSR.
- **Доска как состояние** — карточка проходит весь путь, ничего не теряется.
- **Агент — участник, не магия** — каждое его действие трассируется и обратимо.
- **Человеко-гейт на необратимом** — агент предлагает, человек подтверждает важное.
- **Дешёвый переезд на Postgres** — БД скрыта за `data-access`, миграция не ломает код.

## 3. Архитектура

```
                ┌─────────────────────────────────────────┐
   Telegram ──► │ OpenClaw (агент)                         │
   /Claude      └───────────────┬─────────────────────────┘
                                │ MCP (tools)
                                ▼
   Browser ──► [web SPA] ──REST/WS──► [api: NestJS+Fastify] ──► [Prisma] ──► SQLite file
                                          │
                                          └── Socket.IO ──► живое обновление доски
```

- **apps/api** — NestJS на Fastify-адаптере. REST для CRUD, Socket.IO-gateway для realtime,
  бизнес-логика (жизненный цикл карточки, запись Activity), раздача статики `web`.
- **apps/web** — Vite + React SPA. Канбан, drag&drop, карточка с тредом комментариев,
  переключение view «Идеи» / «Разработка». Подписка на Socket.IO.
  - **UI-кит: shadcn/ui.** Стартовый пресет — `b6GNZujOi`:
    - инициализация: `--preset b6GNZujOi`
    - ссылка: https://ui.shadcn.com/create?preset=b6GNZujOi
- **apps/mcp** — отдельный MCP-сервер (Node). Тонкая обёртка над REST API: выставляет
  агенту инструменты (см. AGENT_INTEGRATION.md). Может работать в том же контейнере.
- **libs/shared-types** — TS-типы и DTO, общие для api / web / mcp. Один источник правды.
- **libs/data-access** — Prisma client + репозитории. Абстракция БД (SQLite сейчас, Postgres потом).

## 4. Модель данных (черновик)

```
User      { id, email, name, passwordHash, role, createdAt }
ApiToken  { id, userId, name, tokenHash, scope, lastUsedAt, createdAt }   // для агента
Board     { id, name, createdAt }
Column    { id, boardId, key, title, order }   // key: idea|triage|backlog|in_dev|done
Card      { id, boardId, columnKey, title, body, priority, order,
            createdBy, assignee, createdAt, updatedAt }
Comment   { id, cardId, authorType, authorId, body, createdAt }  // authorType: human|agent
Label     { id, boardId, name, color }
CardLabel { cardId, labelId }
Activity  { id, cardId, actorType, actorId, action, before, after, createdAt } // трейс/откат
```

`authorType` / `actorType` явно различают человека и агента — для UI-маркеров и аудита.

## 5. Жизненный цикл карточки

Один набор статусов, две «доски» = два сохранённых view:

```
idea ──► triage(разбор) ──► backlog ──► in_dev ──► done
└──────── view «Идеи» ────────┘ └──────── view «Разработка» ──────┘
```

- **«Перенос в разработку»** = смена `columnKey` с `triage`/`backlog`, **один API-вызов**.
  Комментарии, разбор и история остаются на карточке.
- View — это фильтр по статусам поверх общего набора, не отдельная сущность.

## 6. Поток «идея → разработка»

1. Пользователь (или агент по сообщению в мессенджере) создаёт карточку в `idea`.
2. Агент разбирает: задаёт вопросы, ресёрчит, пишет резюме/оценку в комментарии,
   проставляет лейблы → двигает в `triage`.
3. Решение «берём» → карточка в `backlog`/`in_dev`, агент дописывает мини-спеку в `body`.
4. Все мутации агента пишутся в `Activity` (можно посмотреть и откатить).

## 7. Auth

- Email + пароль, JWT. Старт — один пользователь / небольшая команда.
- **API-токены** (`ApiToken`) для агента — отдельная сущность со скоупом и отзывом,
  чтобы не выдавать агенту пользовательскую сессию.

## 8. Деплой и ресурсы

- **Один контейнер**: api раздаёт собранный `web` (статика) + REST + WS; mcp рядом.
- SQLite-файл — на смонтированном volume (персистентность + простой бэкап копией файла).
- `docker-compose.yml`: сервис `moongatracker` + volume под БД. Без отдельного nginx/Postgres.
- Ориентир: **~120–180 МБ RAM**, поверх n8n/OpenClaw влезает в текущий сервер.

## 9. Нефункциональное

- **Бэкап**: cron-копия SQLite-файла (можно через уже имеющийся n8n).
- **Трейс**: каждое действие агента — в `Activity` с `before/after`.
- **Откат**: эндпоинт «revert activity» восстанавливает `before`.
- **Масштаб**: при росте до многих одновременных писателей — миграция на Postgres через
  Prisma (код не меняется благодаря `data-access`).

## 10. Явно вне scope (YAGNI на старте)

Свой SSO/LDAP, многотенантность, мобильное приложение, продвинутая аналитика/дашборды,
автоматизации внутри доски (это делает n8n/агент снаружи).
