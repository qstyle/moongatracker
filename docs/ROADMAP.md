# Moongatracker — роадмап

Поэтапно, каждый этап — самостоятельная ценность. Детальный план реализации
составляется отдельно (следующий шаг после согласования дизайна).

## Phase 0 — каркас монорепо
- Nx workspace, `apps/api` (NestJS+Fastify), `apps/web` (Vite+React), `libs/shared-types`, `libs/data-access`.
- Prisma + SQLite, первая миграция, базовая схема (User, Board, Column, Card).
- Dockerfile + docker-compose (один контейнер, volume под SQLite).
- **Готово, когда:** `nx serve` поднимает api+web локально, пустая доска рендерится.

## Phase 1 — MVP канбана (человек)
- CRUD доски/колонок/карточек, drag&drop, комментарии, лейблы.
- Auth (email+пароль, JWT).
- Два view: «Идеи» / «Разработка» поверх статусов.
- Socket.IO — живое обновление.
- **Готово, когда:** можно вести идею→разработку руками в браузере.

## Phase 2 — агент-слой
- `ApiToken` + скоупы.
- `Activity`-трейс на все мутации + откат.
- `apps/mcp` с набором tools (см. AGENT_INTEGRATION.md).
- **Готово, когда:** Claude/OpenClaw создаёт и двигает карточки через MCP, действия видны в истории.

## Phase 3 — поток с OpenClaw
- Скилл/конфиг OpenClaw под доску (создание идей из мессенджера, разбор, перенос).
- Человеко-гейт на необратимое.
- Бэкап SQLite через n8n.
- **Готово, когда:** «идея из Telegram → разбор с агентом → в разработку» работает end-to-end.

## Phase 4 — полировка (по необходимости)
- Фильтры/поиск, приоритеты, базовая статистика.
- При росте нагрузки — миграция на Postgres через `data-access`.

## Принципы по ходу
- YAGNI: не тащим SSO/мультитенант/дашборды на старте.
- Каждый этап деплоится и проверяется на реальном сервере до следующего.
