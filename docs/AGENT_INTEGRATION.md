# Moongatracker — интеграция с агентом

Как AI-агент (OpenClaw в мессенджере, Claude Code, n8n-флоу) работает с доской.

## Принцип

Агент не имеет прямого доступа к БД. Он ходит через **REST API** по **API-токену**, а для
удобного «инструментального» доступа поверх API поднят **MCP-сервер** (`apps/mcp`).
Каждая мутация фиксируется в `Activity` — это трейс и точка отката.

## Аутентификация агента

- Сущность `ApiToken` (см. `prisma/schema.prisma`): `name`, `scopes String[]`, `tokenHash`, `revokedAt`.
- Токен передаётся в `Authorization: Bearer <token>`.
- Скоупы: `cards:read`, `cards:write`, `comments:write`. **Пока не энфорсятся** —
  хранятся и видны в аудите, но доступ по ним не ограничивается (см. бэклог: энфорс скоупов).
- Токен агента **не** является пользовательской сессией — отдельный субъект в аудите
  (`actorType = agent`). Токен привязан к пользователю-владельцу и даёт доступ ко всем его проектам.

## MCP-инструменты (apps/mcp)

Тонкая обёртка над REST. Актуальный набор tools:

| Tool | Назначение |
|------|-----------|
| `list_projects` | проекты, доступные токену |
| `list_boards` | доски проекта |
| `list_cards` | карточки доски (опц. фильтр по `columnId`) |
| `list_my_cards` | инбокс: карточки, назначенные на этого агента, по всем доскам |
| `get_card` | полная карточка **с комментариями и историей** (по id / boardId+number / key+projectId) |
| `create_card` | создать карточку (`boardId`, `columnId`, `title`, `body?`, `priority?`) |
| `update_card` | заголовок/тело/приоритет/колонка/исполнитель (`assigneeType`+`assigneeId`, `assigneeId:"me"` — самоназначение) |
| `assign_card` | забрать карточку себе (`me`) или снять исполнителя (`none`) |
| `move_card` | сменить `columnId` (колонки свободные, фиксированного набора статусов нет) |
| `comment_card` | добавить комментарий (`authorType=agent`) |
| `list_comments` | тред комментариев карточки |
| `list_activity` | история действий по карточке |
| `propose_card_deletion` | запросить удаление карточки (человеко-гейт — не удаляет напрямую) |
| `list_wiki`, `get_wiki_page`, `create_wiki_section`, `create_wiki_page`, `update_wiki_page` | вики |
| `get_canvas` | чтение canvas (запись агенту пока запрещена) |

`move_card` и `update_card` — это и есть «перенос идеи в разработку» одним вызовом.
Лейблы/теги в текущей схеме отсутствуют (были удалены), поэтому tools их не принимают.

## Трейс и откат

- Любая мутация пишет `Activity { action, before, after, actorType, actorId }`.
- UI помечает действия агента (значок «agent») и показывает историю карточки.
- Эндпоинт отката восстанавливает `before` для выбранной записи `Activity`.

## Человеко-гейт

Необратимое или «дорогое» агент **предлагает**, а не делает молча. Это реальная механика
(сущность `Proposal`), а не только соглашение:

- **Удаление карточки** гейтнуто: у агента нет прямого delete — он вызывает
  `propose_card_deletion`, создаётся `Proposal(status=pending)`, а владелец проекта
  подтверждает или отклоняет — в UI (Настройки → «Согласования») или прямо в Telegram
  (inline-кнопки). Удаление применяется только после approve.
- Прочие «дорогие» действия (напр. массовые операции) могут быть добавлены в гейт по мере
  надобности через ту же модель.

## Сценарий «разбор идеи» (end-to-end)

```
1. Пользователь в Telegram: «идея: автотегирование отзывов по тону»
2. OpenClaw → create_card(title=..., columnId=<колонка «Идеи»>)
3. OpenClaw разбирает, задаёт вопросы в чате, пишет резюме:
   → comment_card(...), update_card(body=...), move_card(<колонка «Разбор»>)
4. Решение «берём» → move_card(<колонка «Бэклог»>), update_card(body="мини-спека...")
5. Всё действие видно в list_activity, при ошибке — откат.
```

## Связь с n8n

n8n остаётся для **детерминированной** части (расписания, вебхуки, бэкап SQLite,
уведомления). «Думающие» шаги делегируются агенту через те же REST/MCP.
